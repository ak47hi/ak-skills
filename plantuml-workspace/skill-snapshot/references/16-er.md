# ER (entity-relationship) diagrams

Shows a relational data model: entities (tables), their attributes, and the cardinality of relationships between them. Use when the question is about *schema*: tables, columns, primary keys, foreign keys.

For OO domain models with methods and inheritance, use Class (`references/12-class.md`).

PlantUML implements ER as an extension of class diagrams, using the `entity` keyword and crow's-foot (information engineering) cardinality notation.

## Entities

```puml
entity Customer {
    * id : uuid <<generated>>
    --
    * email : varchar
    * created_at : timestamp
    name : varchar
    deleted_at : timestamp
}
```

Conventions:
- `*` prefix marks an attribute as **NOT NULL / required**. The space after `*` matters (avoids being parsed as bold).
- `--` separator after the primary key visually divides PK from the rest.
- `<<generated>>` for auto-generated values (uuid, sequence, identity).
- Always include the type after `:` (`uuid`, `varchar`, `int`, `timestamp`, `boolean`, etc.).

## Foreign keys

```puml
entity Order {
    * id : uuid <<generated>>
    --
    * customer_id : uuid <<FK>>
    * status : varchar
    * total_cents : int
    * created_at : timestamp
}
```

`<<FK>>` flags the column as a foreign key. The relation line shows what it references.

## Cardinality (crow's foot)

Crow's-foot notation is the PlantUML default for ER. Read the notation closest to each entity:

| Notation | Meaning at this end |
|---|---|
| `\|\|` | Exactly one |
| `\|o` | Zero or one |
| `}\|` | One or many |
| `}o` | Zero or many |

Combine the two ends with `--` (or `..` for non-identifying):

```puml
Customer ||--o{ Order      ' one customer has zero or many orders
Order ||--|{ LineItem      ' one order has one or many line items
Order }o--|| PaymentMethod ' many orders use exactly one payment method
```

Reading "left-to-right": Customer (exactly one) — has zero-or-many — Order.

## Identifying vs non-identifying

- Solid line (`--`): identifying relationship — the child's PK includes the parent's FK.
- Dashed line (`..`): non-identifying — the FK is a regular column.

Most modern schemas use non-identifying relations (`..`). Use `--` only if the schema really uses composite PKs.

## Layout

For dense ER diagrams, orthogonal connectors are easier to read:

```puml
skinparam linetype ortho
```

For wide schemas:

```puml
left to right direction
```

## Notes

```puml
note right of Order : status ∈ {DRAFT, SUBMITTED, PAID}
```

## Indexes / constraints (informal)

PlantUML doesn't model indexes natively. If you need to convey them, use a note:

```puml
note bottom of Customer : index(email) unique
```

## Anti-patterns specific to ER

- **No cardinality.** Every relation needs crow's-foot notation. A line with no cardinality is just decoration.
- **Everything required.** Marking every column with `*` is no information. Use `*` only for genuinely NOT NULL columns.
- **No `<<FK>>` on foreign keys.** The reader has to trace lines to figure out which column is the FK. Mark it.
- **No `<<generated>>` on surrogate keys.** Telling the reader the PK is system-generated vs application-provided matters.
- **Method-like attributes.** Don't write `findById()` in an ER diagram. ER is data; methods belong in Class.
- **Class arrows.** `<|--`, `*--`, `o--` are class-diagram arrows. ER uses crow's foot.
