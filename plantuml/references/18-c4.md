# C4 diagrams

Generates C4 model diagrams using the [C4-PlantUML stdlib](https://github.com/plantuml-stdlib/C4-PlantUML). Use when the user asks for C4 specifically, or when the audience is mixed (engineers + product + leadership) and conventional UML component diagrams would be too noisy.

C4 has four levels. **Don't mix levels in one diagram** — that's the most common mistake. See "Abstraction-level discipline" below.

## The four levels

| Level | Question it answers | Template |
|---|---|---|
| Context | Who uses our system, and which external systems does it touch? | `templates/c4-context.puml` |
| Container | What apps / services / datastores make up our system, and how do they communicate? | `templates/c4-container.puml` |
| Component | Inside ONE container, what are its components and how do they collaborate? | `templates/c4-component.puml` |
| Dynamic | For ONE scenario, what's the runtime call sequence across containers? | `templates/c4-dynamic.puml` |

Pick one. Pick correctly — Container is by far the most useful for general architecture diagrams; Context is good for executive-level; Component is for deep internal work; Dynamic is for explaining specific flows to engineers.

## Including the stdlib

**Primary (PlantUML standard library, bundled with official PlantUML since ~2020):**

```puml
!include <C4/C4_Context>
!include <C4/C4_Container>
!include <C4/C4_Component>
!include <C4/C4_Dynamic>
```

Include exactly **one** — whichever matches the diagram level. The later includes transitively include the earlier ones, so `C4_Container` already has Context's macros.

**Fallback (direct GitHub URL — use if the user's PlantUML build doesn't ship the stdlib):**

```puml
!include https://raw.githubusercontent.com/plantuml-stdlib/C4-PlantUML/master/C4_Context.puml
!include https://raw.githubusercontent.com/plantuml-stdlib/C4-PlantUML/master/C4_Container.puml
!include https://raw.githubusercontent.com/plantuml-stdlib/C4-PlantUML/master/C4_Component.puml
!include https://raw.githubusercontent.com/plantuml-stdlib/C4-PlantUML/master/C4_Dynamic.puml
```

Default to the short form. Document the URL form in a comment in the generated `.puml` so users on older builds know the fallback.

## Macros

### People and systems (Context, all levels)

```
Person(alias, "label", "description")
Person_Ext(alias, "label", "description")

System(alias, "label", "description")
System_Ext(alias, "label", "description")
SystemDb(alias, "label", "description")
SystemQueue(alias, "label", "description")
```

`_Ext` variants render in muted/greyed style to mark systems/people *we don't own*. Use it consistently — mislabeling ownership is a common error.

### Containers (Container level)

```
Container(alias, "label", "technology", "description")
ContainerDb(alias, "label", "technology", "description")
ContainerQueue(alias, "label", "technology", "description")
Container_Ext(alias, "label", "technology", "description")
```

The **technology** label is what makes C4 Container useful — "API (Go, gRPC)" vs just "API". Always include it.

### Components (Component level)

```
Component(alias, "label", "technology", "description")
ComponentDb(alias, "label", "technology", "description")
```

Components live *inside* one container. Don't draw multiple containers' components in the same diagram — split.

### Relationships

```
Rel(from, to, "label", "technology")
Rel_D(from, to, "label", "technology")
Rel_U(from, to, "label", "technology")
Rel_L(from, to, "label", "technology")
Rel_R(from, to, "label", "technology")
Rel_Back(from, to, "label", "technology")
BiRel(from, to, "label", "technology")
```

`_D/U/L/R` variants force layout direction (down/up/left/right). Use sparingly to fix layout, not for semantic meaning.

**Label every `Rel`** with what the relationship does ("Reads from", "Authenticates via", "Publishes events to") and (optionally) the protocol.

### Boundaries

```
System_Boundary(alias, "label") { ... }
Container_Boundary(alias, "label") { ... }
Enterprise_Boundary(alias, "label") { ... }
Boundary(alias, "label", "type") { ... }
```

Boundaries are visual groupings — "our system" vs "external", "this bounded context" vs "that one".

### Layout helpers

```
Lay_D(from, to)
Lay_U(from, to)
Lay_L(from, to)
Lay_R(from, to)
LAYOUT_LEFT_RIGHT()
LAYOUT_TOP_DOWN()
LAYOUT_WITH_LEGEND()
SHOW_LEGEND()
```

`LAYOUT_WITH_LEGEND()` or `SHOW_LEGEND()` adds a legend showing person / system / container styles. Useful for diagrams shared with non-developers.

## Abstraction-level discipline

The single most important rule. Each level shows exactly what its name says — no more.

**Context** shows:
- The system being described.
- People who use it.
- External systems it interacts with.
- Nothing inside the system. No containers. No components.

**Container** shows:
- The apps, services, datastores **inside the system**.
- External systems and people **at the boundary**.
- Nothing about the inside of any container.

**Component** shows:
- The components **inside ONE container**.
- The other containers as boundaries / endpoints — not their internals.
- Not the people. Not external systems (except as endpoints).

**Dynamic** shows:
- A numbered sequence of calls across containers (or components).
- Like a sequence diagram but using C4 elements.

If you find yourself adding "the database is Postgres" to a Context diagram, you're at the wrong level. Use Container.

## Dynamic-specific

Dynamic diagrams add numbered calls:

```puml
Rel(web, api, "1. POST /login")
Rel(api, auth, "2. verify")
Rel(auth, db, "3. SELECT")
Rel(db, auth, "4. row")
Rel(auth, api, "5. ok")
Rel(api, web, "6. 200 {token}")
```

Number every `Rel`. The numbering is the diagram's information.

For pure message-exchange flow at finer granularity than containers (e.g. classes or modules), use a Sequence diagram instead — it has richer notation for that level.

## Anti-patterns specific to C4

- **Mixed levels in one diagram.** The single biggest C4 mistake. Each level is exclusive.
- **`System_Ext` confused with `System`.** External systems we don't own use `_Ext`. Mixing them lies about ownership.
- **Container without `technology` label.** `Container(api, "API")` is half-useful. `Container(api, "API", "Go, gRPC")` is the point of C4 Container.
- **`Rel` with empty label.** Every relationship should describe what it does. Empty-labeled `Rel` is a line.
- **Drawing internal components in a Context diagram.** If components belong, you're drawing the wrong diagram.
- **No `Person`.** Most Context and Container diagrams should include at least one `Person` (the user). A C4 diagram of a system with no human user might be a deployment diagram.
